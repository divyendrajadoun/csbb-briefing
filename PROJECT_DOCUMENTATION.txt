================================================================================
  1. PROJECT OVERVIEW
================================================================================

Palini Ji is a bilingual (Hindi/English/Hinglish) AI briefing assistant built
for the Office of the Chief Secretary, Bihar. It provides:

- Voice + chat interface with a HeyGen live avatar
- Grounded answers using 7 tool calls (schedule, schemes, alerts, files,
  officers, approvals, knowledge base search)
- Role-based access (Chief Secretary, Principal Secretary, Support Cell)
- File upload with OCR for handwritten notes, PDFs, and images
- Per-user configurable API keys (Groq, ElevenLabs, HeyGen)
- Email forwarding of any bot response
- Full audit logging (append-only JSONL)

================================================================================
  4. BACKEND FILES - DETAILED DESCRIPTION
================================================================================

--- main.py ---
  The FastAPI application entry point. Contains:
  - GET /health          : Health check endpoint
  - GET /roles           : Returns available roles (cs, ps, support_cell)
  - POST /settings       : Saves per-session API key overrides
  - POST /tts            : Text-to-speech via ElevenLabs (uses per-session keys)
  - GET /contacts        : Returns officer contacts for email modal
  - POST /send-email     : Sends email via Resend API
  - POST /upload         : Uploads file to knowledge base (PDF/image/text)
  - WebSocket /ws/{role} : Real-time chat with tool-calling LLM

  Manages three in-memory dictionaries:
  - sessions: dict[str, list]          # conversation history per session
  - knowledge_bases: dict[str, list]   # uploaded document chunks per session
  - session_settings: dict[str, dict]  # API key overrides per session

  All three are cleaned up on WebSocket disconnect.

--- config.py ---
  Loads environment variables from .env and defines constants:
  - GROQ_API_KEY, ELEVENLABS_API_KEY, ELEVENLABS_VOICE_ID
  - HEYGEN_API_KEY, RESEND_API_KEY, RESEND_FROM_EMAIL
  - MODEL = "meta-llama/llama-4-scout-17b-16e-instruct"
  - VISION_MODEL = "llama-4-scout-17b-16e-instruct"
  - MAX_TOKENS = 4096
  - MAX_UPLOAD_MB = 10
  - DATA_DIR, AUDIT_LOG_PATH

--- claude_client.py ---
  Handles LLM interaction with Groq API:
  - process_message() accepts user text, role, conversation history,
    session_id, and optional groq_api_key_override
  - Builds messages with system prompt + conversation history
  - Runs a tool-use loop: sends to Groq, executes tool calls, loops
    until the model produces a final text response
  - Retry logic for tool_use_failed errors (2 retries, then fallback to no tools)
  - Sends events via callback: "thinking", "tool_call", "message_complete"

--- tools.py ---
  Defines 7 tools with OpenAI-format schemas and handlers:

  1. get_schedule(date, date_end)
     - Returns CS calendar events for a date or date range
     - Source: CS Calendar System

  2. get_scheme_status(scheme_id)
     - Returns status, metrics, lagging districts for government schemes
     - Supported: mgnrega, pmay_g, jjm, jeevika, midday_meal
     - Source: Scheme MIS Dashboard

  3. get_alerts(severity)
     - Returns situational alerts sorted by severity
     - Types: weather, flood, health, SLA breach, law & order, infrastructure
     - Source: State Situation Room

  4. get_pending_files(priority, department)
     - Returns files pending CS approval with ageing info
     - Restricted to CS and PS roles
     - Source: CS File Tracking System

  5. get_officer_profile(query)
     - Looks up officers by name, post, or district
     - Returns service details, contact, posting history
     - Restricted to CS and PS roles
     - Source: HR-MIS, GAD

  6. submit_for_approval(action_type, title, details)
     - Queues an action for human approval (NOT auto-executed)
     - Types: meeting_cancellation, mom_draft, broadcast
     - Restricted to CS and PS roles
     - Source: Workflow Engine

  7. search_knowledge_base(query, session_id)
     - Keyword search across uploaded document chunks
     - Returns top 5 matching chunks with filename and page number
     - Available to all roles
     - Source: Knowledge Base (uploaded docs)

  execute_tool(name, arguments, session_id) dispatches calls to handlers.

--- roles.py ---
  Defines three roles with scoped tool access:

  - cs (Chief Secretary): ALL 7 tools
  - ps (Principal Secretary): ALL 7 tools
  - support_cell (Support Cell): get_schedule, get_scheme_status,
    get_alerts, search_knowledge_base (4 tools, no files/officers/approvals)

--- system_prompt.py ---
  Defines the Palini Ji persona and behavioral guidelines:
  - Always use tools before answering factual questions
  - Cite sources inline
  - Be concise (CS time is valuable)
  - Flag severity for alerts
  - Bilingual responses matching user language
  - Role awareness and access restrictions
  - Approval actions are HELD, not executed
  - No hallucination policy
  - Knowledge base: use search_knowledge_base, cite filename and page

  Dynamic context: today's date, user role, KB status (chunk count).

--- kb.py ---
  Knowledge base extraction logic:

  - extract_text_from_upload(filename, contents, content_type)
    Routes by file type:
    * PDF: PyMuPDF extracts native text per page. Scanned/handwritten
      pages (< 50 chars of text) are rendered to image at 200 DPI
      and OCR'd via Groq vision model.
    * Images (PNG/JPG/WEBP): Direct OCR via Groq vision model.
    * Text files (TXT/MD): Read as UTF-8.

  - _ocr_image(image_bytes, mime)
    Sends base64-encoded image to Groq vision model with prompt:
    "Extract ALL text from this image exactly as written.
     The text may be in Hindi, English, or both.
     Return only the extracted text, nothing else."

  - _extract_pdf(contents)
    Opens PDF with PyMuPDF, iterates pages, extracts text or falls
    back to vision OCR for scanned pages.

--- audit.py ---
  Append-only audit logging to audit.jsonl:
  - log_event(event_type, role, session_id, data)
  - Events: user_message, assistant_response, file_uploaded, upload_error,
    email_sent, email_error, disconnect, error

--- hooks/useWebSocket.js ---
  Custom React hook for WebSocket connection:
  - connect(): Opens WebSocket to /ws/{role}
  - disconnect(): Closes WebSocket
  - sendMessage(text): Sends user_message JSON
  - Handles events: connected, thinking, tool_call, message_complete, error

  Custom React hook for voice input + TTS output:
  - startListening(onResult): Web Speech API with lang="hi-IN"
  - stopListening(): Stops recognition
  - speak(text, sessionId): Calls /tts endpoint, plays audio
  - unlockAudio(): Unlocks iOS/Safari audio playback on first interaction
  - Returns: isListening, transcript, startListening, stopListening, speak, unlockAudio

================================================================================
  9. LLM MODEL DETAILS
================================================================================

MODEL: meta-llama/llama-4-scout-17b-16e-instruct (via Groq)
  - 500K tokens per day limit (vs 100K for llama-3.3-70b)
  - ~0.4 second latency per call
  - Full tool calling (function calling) support
  - Supports Hindi + English

VISION MODEL: llama-4-scout-17b-16e-instruct (via Groq)
  - Used for OCR of scanned/handwritten pages
  - Accepts base64-encoded images
  - Handles Hindi + English handwriting

Previous models tried and rejected:
  - llama-3.3-70b-versatile: Hit 100K TPD rate limit
  - compound-beta-mini: Does not support tool calling
  - qwen/qwen3-32b: Supports tools but ~35 second latency (too slow)
